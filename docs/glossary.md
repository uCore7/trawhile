# Glossary

Terms are listed alphabetically. Where a term appears in requirements or architecture documents, it is used with the meaning defined here.

---

**Anonymization** Irreversible removal of a user's personal data: the `user_profile` row is deleted (cascading to `user_oauth_providers`, `quick_access`, and `mcp_tokens`), while the `users` stub and all `time_entries` are retained so that historical aggregates remain intact. The anonymised user can re-register only with a new invitation, which creates a new, unlinked identity. See SR-047.

**Authorization level** One of three permission levels (`view`, `track`, `admin`) that can be granted to a user on a node. They form a strict subset hierarchy: `view` ⊂ `track` ⊂ `admin`. A grant of a higher level implies all lower levels. See also: *effective authorization*, *node authorization*.

**Bootstrap** The one-time process of creating the first System Admin. Because no admin can exist before any user logs in, the `BOOTSTRAP_ADMIN_EMAIL` environment variable designates the email address that receives `admin` on the root node on first OIDC login. See SR-001.

**Effective authorization** The highest authorization level a user holds on a specific node, considering both direct grants on that node and grants on any ancestor node. Resolved via recursive CTE (Q2 in schema.sql). A grant on an ancestor is effective on all descendants — this is the core authorization model.

**Freeze cutoff** A rolling point in time computed as `NOW() − freeze-offset-years`. Time entries with `started_at` before this date are immutable: they cannot be edited or deleted by anyone, including System Admins. The offset is set in the system config file (`trawhile.freeze-offset-years`).

**Leaf node** A node that has no active children. Leaf nodes are the only nodes against which time can be tracked. A node becomes a leaf when all its children are deactivated or deleted; it ceases to be a leaf when a child node is created or reactivated.

**MCP (Model Context Protocol)** An open protocol that allows AI assistants and other clients to invoke structured tools on a server. trawhile implements an MCP server that exposes time tracking data (time entries, node tree, member summaries) to authorized MCP clients using Bearer token authentication.

**MCP token** A long-lived secret generated by a User to authorize an MCP client to act on their behalf. Only the SHA-256 hash is stored; the raw token is shown once and never retrievable. Tokens can have an optional expiry and are revocable at any time.

**Node** The fundamental organizational unit of trawhile's hierarchy. A node can represent anything the company chooses to track time against: a project, work package, task, team, absence type, etc. Nodes form a tree; the root node is the unique top-level ancestor seeded by Flyway. Each node has a name, optional description, color, icon, and logo. See also: *leaf node*.

**Node authorization** A grant of a specific *authorization level* (`view`, `track`, or `admin`) to a specific user on a specific node. Stored in the `node_authorizations` table. One user may hold at most one authorization per node (stored as the highest granted level). See also: *effective authorization*.

**OIDC (OpenID Connect)** An identity layer built on top of OAuth 2.0. trawhile delegates all authentication to configured OIDC providers (Google, Apple, Microsoft Entra ID, Keycloak) and stores only the provider name and the provider's subject ID — never an email address or password.

**Operator** The person responsible for deploying and maintaining the trawhile instance (Docker Compose, config file, backups). In practice this is typically the same person as the System Admin (ST-4).

**Pending invitation** A record created when a System Admin invites a person by email address. Contains the email and is linked to a pre-created `users` row. Deleted when the invitee completes their first OIDC login. Expires automatically after 90 days; expiry deletes both the invitation and the associated `users` row.

**Purge** The scheduled batch deletion of activity data (time entries, requests) older than the configured retention period. Runs on operator-configured dates (ANNUALLY, BIANNUALLY, or QUARTERLY). Tracked in the `purge_jobs` table; idempotent — resumes from stored `cutoff_date` on restart. See also: *retention period*.

**Quick access** A user's personal list of up to 9 nodes for fast tracking start, ordered by the user. Nodes remain in the list when deactivated (annotated with a non-trackable flag) and are not removed automatically.

**Request** An in-app structured message submitted by any user with at least `view` on a node, addressed to the Node Admins of that node. Uses a system-defined template (e.g. "Grant authorization", "Create child node") or free text. Can be closed by any Node Admin of the target node or any ancestor.

**Retention period** The configured duration (`trawhile.retention-years`, minimum 2 years) after which activity data (time entries and requests) is permanently deleted by the next scheduled purge. Set in the system config file; cannot be changed via the UI.

**Root node** The unique top-level node in the tree (`parent_id IS NULL`), seeded by Flyway. All other nodes are descendants of the root. A user with `admin` on the root node is a System Admin and has effective authorization over the entire tree.

**SSE (Server-Sent Events)** A standard HTTP mechanism for unidirectional server-to-client push. trawhile uses SSE to push live state updates (tracking changes, node tree changes, authorization changes, request events) to all browser sessions without polling. Each authenticated session maintains one persistent SSE connection.

**System Admin** A User with effective `admin` authorization on the root node. Has full scope over the entire node tree: all Node Admin capabilities across all nodes, plus user management and system settings access. IS-A Node Admin IS-A User. Typically also the person who deployed the instance (operator). See ST-4.

**Tracking** The act of recording time spent against a node. A user may have at most one active tracking session at any time (enforced by a partial unique index on `time_entries` where `ended_at IS NULL`). Tracking is always user-initiated; the system never starts or stops tracking automatically.

**Visitor** An unauthenticated person who can access only the About page. Visitors can read the built-in GDPR data summary and download the SBOM and OpenAPI specification without logging in. See ST-1.
